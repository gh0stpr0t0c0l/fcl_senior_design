\begin{abstract}
This project proposal outlines our intent to reimaginine the Feedback Controls Laboratory experience at Cedarville University. The main goals of this project are to make the lab experience more engaging and relevant for students, simpler to work with, more reliable than previous solutions, and more cost-effective. To achieve these ends, we plan to design a cheap drone platform that is capable of communicating fundamental feedback controls concepts with a simple software user interface. We will start by selecting hardware components that meet our end goals, creating a hardware prototype. Once this prototype is complete, we will program, test, and modify it until we are satisfied with the platform. Finally, we will refine a final design and software solution including simple choices for the user that aid learning. If this is completed with time to spare, we will refine the experience, adding features and focusing the lab experience. Ultimately, we hope to transform the way students view Feedback Controls lab, relating classwork to concepts they are familiar with and making lab time enjoyable.
\end{abstract}

\section{Definition Statement}

This project seeks to redefine the Cedarville Feedback Controls Laboratory experience by designing a quadcopter drone to use as an entertaining platform to teach classroom concepts.

\section{Background}

The current lab experience for EGME-4660 uses the Feedback Analogue Unit 33-110 and the Feedback Mechanical Unit 33-100. Although the labs using the equipment do cover the desired content, they are often tedious and perhaps confusing to new users, especially users unfamiliar with Feedback Controls concepts. The user interface to conduct the experiments is a circuit board with nodes and banana plug cords. The user can connect these nodes to create feedback circuits whose effect is then shown using the mechanical unit. As mentioned, understanding what is happening between the units (and even in the electrical circuits) can be challenging for students. This causes the applications of the class material to  seem boring and irrelevant, especially for mechanical engineering students who may not have extensive electronics background. In addition, the equipment is old and fragile - adjusting the power supply or wiring the power incorrectly can result in damage to the equipment, often requiring parts to be replaced. Since the equipment is quite expensive, the current repair method has been de-soldering and soldering components by hand. We plan to improve the lab experience by replacing the current equipment with cheap and robust drones that can be reprogrammed for the purposes of the lab with a user-friendly interface. In this way, the instructor can choose the complexity of the tasks given to students for the lab, instead of making them start from ground zero.

\section{Objectives}
\subsection{Marketing Requirements}

The following requirements define the end user experience as desired by the customer:
  \begin{itemize}
    \item The final product must focus on autonomous actuation and control.
    \item The laboratory experience must include the ability to implement a PID control design.
    \item The final product must be able to physically and graphically display a step response.
    \item The students must have the opportunity to measure a Bode plot in real life.
    \item The laboratory experience must include the ability to see and correct the effects of physical and observable disturbances and noise.
    \item The final product should showcase the results of controllers physically in real life and not just with numbers and data.
    \item The final product must have the ability to log data to a neutral format independent of real time visualizations.
    \item The software solution must include widget to generate understandable plots of the data.
    \item The final laboratory experience must include the capability to perform a tracking experiment.
    \item The laboratory experience must take relevant safety precautions.
    \item The final platform must be able to take basic commands and log the response. 
  \end{itemize}

\section{Strategy}
This project is very software heavy, but hardware is going to inform software decisions. All members will participate in the selection and planning of hardware and software, but the research will be led by the members who have the most experience and knowledge in those areas. The team aims to select parts that will allow the drone to sense its environment and produce repeatable results in a lab setting.

\subsection{Hardware}
\begin{enumerate}

\item Microcontroller
\begin{itemize}
\item Very important component because other hardware and software selections depend on what microcontroller is chosen
\item Needs to support a good amount of I/O for sensors and motors
\item Needs to be fast to support multiple PID loops
\item Could have onboard networking to support wireless telemetry
\end{itemize}

\item Custom PCB
\begin{itemize}
\item Needs to combine all important components into a simple, pre-assembled package
\item Could be designed to also be drone frame
\end{itemize}

\item Inertial Measurement Unit (IMU)
\begin{itemize}
\item Needs to accurately sense rotation in 3 axes for calculating error
\end{itemize}

\item Sensors
\begin{itemize}
\item Needs to accurately sense distance in 3 axes
\item Needs to be simple to get data from. IR and ultrasonic sensors are often good for this
\end{itemize}

\item Motors and Motor Controllers
\begin{itemize}
\item Need to be powerful but with few components
\item If the drone kits are small enough, brushed motors would be a good choice
\end{itemize}

\item Battery
\begin{itemize}
\item Needs to be big enough for at least 5 minutes of flight
\item Needs to be easily swappable
\end{itemize}

\item Frame
\begin{itemize}
\item Needs to hold the battery, motors, and PCB in a way to prevent them from damage
\item Needs to be cheap to 3D print
\end{itemize}
 

\end{enumerate}


\subsection{Software}
\begin{enumerate}
\item FreeRTOS
\begin{itemize}
\item FreeRTOS is a real time operating system that is build and optimized for microcontrollers. An RTOS is a framework for switching between many processes.
\item This will allow the system to constantly check sensors, log data, and run the flight logic all at the same time
\end{itemize}

\item Desktop Application
\begin{itemize}
\item Needs to have interface for writing new configurations for the drones
\item Needs to be user friendly
\item Might include real time data visualization
\end{itemize}

\item Flight Telemetry 
\begin{itemize}
\item Whether the flight data is being logged onboard the drone or being streamed to a laptop, it needs to be saved in an easy to use format like CSV
\end{itemize}
\end{enumerate}

\section{Testing Plan}
Since the end goal of this project to create lab environment for students, the tests can be broken down into two sections: construction tests and lab tests.

\subsection{Construction Tests}
These are system tests with the purpose of checking if the various parts of the drone are working together properly.

\begin{enumerate}
	\item \textit{Motor System} - Demonstrate the microcontroller driving multiple motors using a MOSFET circuit
	\item \textit{Onboard Data Logging} - Demonstrate the logging of relevant data in non-volatile memory 
	\item \textit{Wireless Data Logging} - Demonstrate the transmission of relevant data to a nearby laptop
	\item \textit{Distance Sensor Calibration} - Demonstrate the sensors accurately logging distance information
	\item \textit{IMU} - Demonstrate the IMU logging rotation information for calculating error
	\item \textit{PID Loop} - Demonstrate the PID loop functioning correctly by flying the drone
\end{enumerate}

\subsection{Lab Tests}
These tests are to verify that the drone and surrounding systems are able to fulfill the marketing requirements and be useful in the lab setting.

\begin{enumerate}
	\item \textit{Hold Altitude} - Demonstrate the drone holding at an arbitrary target altitude
	\item \textit{Step Response} - Demonstrate the drone changing its target altitude and settling at the new one
	\item \textit{Sinusoidal Response} - Demonstrate the drone changing its height corresponding to an input function
	\item \textit{Constant Disturbance Response} - Demonstrate the drone maintaining a position while a constant disturbance like a fan is applied to it
\end{enumerate}
